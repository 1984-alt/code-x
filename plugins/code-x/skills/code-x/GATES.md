# Code-X V1 — GATES

> Load-on-demand (NOT read every session). The kernel points here when a gate fires. Read only the gate you're at. Each gate = a rule + how it's mechanically checked.

## G1 · Card Compilation Gate — the most important safety gate
Because a builder reads only ONE card, the card must be correct or the whole small-context system is unsafe (a bad card is more dangerous than a long packet — it looks clean while silently omitting an invariant).
- **Compile:** the Card Compiler (top model) reads the locked packet **once** → emits the whole work-order deck (`MODE-A-001`, `BUILD-001…N`, `FINAL-READY`). The builder never creates scope; **the compiler compresses scope.**
- **Audit:** the opposite-family Card Auditor (top model) checks each card against its claimed packet slice. The card's `source_map` names that exact slice (packet id + **frozen packet hash** + sections + requirement ids), so the audit is *traceable*, not vibes. The locked packet is **frozen once cards are cut from it** — no edit-in-place, or the hash breaks. **This audit IS the pre-build cross-family checkpoint — do not stack a second full cross-family review on top.**
- **Cloud-safe:** if a card is sent to a cloud reviewer, its `source_map`/text must be scrubbed first (no private labels/amounts) — see `LESSON-PRIVACY-001`.
- **Shape-check:** `cx check card` validates each card mechanically AND **rejects** it if any of: field missing · allowed/forbidden files undeclared · read-budget over cap · `security_tripwire` unset · `model_tier` unnamed · **`source_map` missing/untraceable** · **`actor_record.cross_review.family == executor.family`** (same-family review is not cross-family — a CEO-authorised `family_substituted` stand-in is allowed but opens a final-ready-blocking `CROSS_FAMILY_RECHECK_PENDING` finding). *(These fields are real gates, not paperwork.)*
- **Deck coverage (reverse):** the locked packet carries a `requirements-manifest.yaml` INSIDE the frozen hash — every requirement id + disposition (`BUILDING | NOT_BUILDING | NOT_APPLICABLE | CEO_DEFERRED`). `cx check deck` proves every BUILDING requirement landed in ≥1 card, every NOT_BUILDING/CEO_DEFERRED carries a `ceo_decision_ref`, no card claims a ghost requirement, and the recomputed packet hash still matches every card. Forward traceability is necessary but NOT sufficient — this closes the reverse direction (packet→deck), so a requirement cannot silently vanish at compile. [RULE:deck-reverse-coverage]
- **PASS = all four.** A failing card → FIX-FIRST → recompile. *AI compiles · opposite AI audits · checker validates shape · deck check proves coverage.*

## G2 · Security — baseline once + tripwire every card
- **Baseline (once, in planning):** one full security/privacy review of the architecture (authn/authz, secrets, PII, input boundaries, retention). Secrets by NAME only.
- **Tripwire (every card):** the compiler SETS the 7-question `security_tripwire` from the packet + `allowed_files`; `cx check` verifies it against the real diff (**not** builder self-attestation). ANY `yes` → **security delta** (a focused security review of just that card's diff).
- **Ship:** one full security/privacy closeout before final-ready.

## G3 · Capsule anti-staleness
After a module is CEO-approved it's frozen into a `MODULE-CAPSULE`. Then:
- **Touch a module → refresh its capsule in the SAME wave**, or the wave cannot clear.
- **The capsule's `commands_to_regression_test` must still pass**, or the wave cannot clear.
Module N reads only its **dependency capsules** — never modules 1…N-1 in full. *(This is what keeps modular cheap.)*

**Fixture-purge semantics (PROP-019, folded 2026-06-12): `DESIGN_FIXTURE_PURGE != DESIGN_DELETE`.** Purging a fixture means replacing fixture DATA inside the CEO-accepted shell — deleting or rebuilding an accepted screen is **P1 drift** (same class as golden-master drift; a real scar from a prior engine build wave). Every engine wave touching a Mode A screen requires:
- a **`fixture_replacement_map`** row per purged screen: `{purged_fixture, live_route, live_template, locked_reference, data_fn_preserved, controls_preserved, screenshot_comparison, dom_marker_comparison}` — no replacement map, no G3 pass;
- a **design-fidelity check** before any model review of a UI module: deterministic comparison of the live DOM snapshot/accessibility tree against the locked reference's **`ui_marker_manifest`** (required regions · required `data-fn` controls · required shell selectors · forbidden_missing), PLUS a screenshot for viewport/visible-shell/no-blank/no-overflow support proof. Marker manifest + DOM = the deterministic gate; screenshots = supporting evidence; **pixel-AI is never the primary gate**;
- the card itself carries **`ui_contract`** (locked artifacts: taste lock · design golden master · Mode A ceo-accept + screenshots + route map + click-path contract; `preservation_rule: replace fixture data, preserve approved shell and controls`). A card that touches templates/static/HTML-rendering routes or deletes Mode A fixture files WITHOUT `ui_contract` in its read set = **P1 — STOP, the card is wrong** (kernel rule). UI review dispatches add the additive `visual_contract_context` block + a **VISUAL CONTRACT CONTINUITY** leg (PROP-015 stanza itself stays verbatim).

MARKER-STUB: hidden/role-less/label-less marker stuffing cannot read green — GPT cross-review 2026-06-12).*

## G4 · Review modes — right-sized (FULL is rare)
| Mode | Reads | When |
|---|---|---|
| **SCAN** | the diff only | tiny / low-risk change |
| **DELTA** | diff + directly-touched files | **the default** |
| **SLICE** | one subsystem | a feature spanning a few files |
| **FULL** | the whole app | ONLY at: build-authorization lock · final-ready/ship · a major security/architecture/privacy change · explicit CEO request |
A FULL review re-reads everything and is the biggest token sink — never trigger it by reflex.
**When** the model review fires is set by G5 (the module boundary); **which** mode it uses is the table above (DELTA by default).

## G5 · One-and-done review loop — review per MODULE, not per card
**deterministic checks run on every card** (free, zero model tokens — `cx check` + consistency lint + tests) so structural/mechanical bugs are caught the moment they appear. The **costly MODEL review (self + cross-family) fires once per MODULE** — when the module is complete and demoable, read as one coherent unit with full context, never as half-built fragments. CEO approval is likewise once per module.
Within that single module review: review once → list ALL findings → fix in ONE batch → verify once → STOP or hand off. **Never** review → fix-one → re-review → repeat. `loop_budget` (default 1) is a hard cap; hitting it = STOP + hand off, do not grind tokens.
**SELF_REVIEW escalation exception (V1.10):** a `review_dispatch.review_kind: SELF_REVIEW` card may run the bounded model-escalation loop — re-review 1/2/3, each coupled to a strictly-upward fixer seat (builder → stronger → strongest via the `fix_cycles` ladder) — so `cx check card` allows `loop_budget.review_fix_cycles ≤ 3` for SELF_REVIEW cards ONLY; CodeRabbit and cross-family stay one-shot (`≤ 1`). This is the ONE deliberate v1.09→v1.10 clause change (the cap, not the seat ladder). Self-review STOP after the cap → escalate to cross-family (the safety valve, not the routine gate). [RULE:self-review-escalation-cap]
**Modules stay small** (minutes to approve) so a batched review never hides much rework.
**Fix-escalation ladder (PROP-021, folded 2026-06-12):** if fix-1 fails review on the SAME finding (stable `finding_id` + `source_review_ref` — same finding = same id; a fix that introduces a new P0/P1 also escalates) or introduces a new P0/P1, the next fix attempt MUST use the next escalation seat defined by `BUILD-ENGINE-PROFILES.fix_escalation` for the active engine; re-dispatching the same seat tier = protocol drift. Fix-3 failure feeds the existing 4-failure cross-family debug rule (the ladder slots BELOW it). Attempt history is **STATE-owned** (`CODE-X-STATE.fix_cycles` rows `{finding_id, attempt_n, seat_ref, result, review_ref}`), never card-owned — cards stay frozen scope contracts and may carry only `fix_policy_ref`. **Escalation does NOT reset `loop_budget.review_fix_cycles`** — it only changes the seat for the next bounded fix attempt on an already-open finding; it is never permission for a new review cycle. [RULE:fix-escalation-ladder]
*✅ BUILT + biting 2026-06-12: `cx check state` validates `fix_cycles` — typed rows · strictly-upward sequential seat ladder per finding · unparseable escalated seat fail-closed · no mid-ladder family switch · no fix-4 row · loop_budget reset rejected — clauses STATE-FIX-CYCLE-SAME-TIER · STATE-FIX-CYCLE-FIX3-NOT-ESCALATED · STATE-FIX-CYCLE-MISSING-FINDING-ID · STATE-FIX-CYCLE-BUDGET-RESET · STATE-FIX-CYCLE-ENGINE-FAMILY (GPT cross-review 2026-06-12: seats bind to the ACTIVE engine's family; `cx check close-turn` re-runs the fix-cycle + incident clauses so state mutated at turn close cannot dodge them).*
**Carve-out — MANDATORY for high-risk foundations (protects catch-rate, the CEO's #1 priority):** a HIGH-RISK foundation card that later cards build on — money / auth / shared data-shape / secrets — **MUST get one checkpoint before any dependent card runs.** The Card Compiler sets `foundation_checkpoint_required: yes` (+ `foundation_checkpoint_reason`) on such a card; when `yes`, `cx check card` **blocks every dependent card until that checkpoint passes**. Risk-tiered: only high-risk foundations trigger it (it is *not* "may" — for those four classes it is required); everything else waits for the normal module-boundary review.

**CEO-demoable boundary (PROP-020 standing rule, CEO ratified 2026-06-12):** cross-family review **NEVER fires unless the module is CEO-DEMOABLE or it is the mandatory foundation checkpoint above.** CEO-demoable is mechanical, not a vibe: `{route_or_command, screenshot_or_log, click_path_or_smoke, ceo_can_verify_without_code: yes}` — UI modules = visible/clickable/runnable by the CEO; backend/headless modules = a command/API/log proof the CEO can verify without reading code. A cross-family dispatch at a non-demoable, non-foundation boundary = protocol drift (a real miss: review was requested when the CEO had nothing to check; Anthropic sessions are scarce and precious).

**Typed deviations + protocol incidents (PROP-020, folded 2026-06-12):**
- **`protocol_deviation`** — any CEO-authorized departure from canonical timing (e.g. cross-family deferred on session limits) is a typed state block `{id, ceo_authorized, ceo_decision_ref, canonical_rule, temporary_rule, blocks: next_card|ship|final_ready|none, review_debt_items[]}` — never blended into state as if normal; review debt stays visible until repaid. `cross_family_boundary: ceo_deferred` REQUIRES `ceo_decision_ref` + `deferred_review_blocks` — and (V1.10) those blocks MUST include `ship`/`final_ready`: a cross-family deferral may **not** permit ship ("last" can never become "never"; cross-family stays the final ship gate). `cross_family_boundary: final` (the CEO-preferred Option B) is fully valid and needs no deferral fields.
- **`review_boundary`** — reviewer taxonomy/timing is state, not prose: `{deterministic_checks_each_card, coderabbit_before_self_review: yes|no|not_applicable, self_review_boundary: card|module|foundation_checkpoint|final, cross_family_boundary: module|foundation_checkpoint|final|ceo_deferred, ...}` — required enums, partial prose rejected.
- **`PROTOCOL_INCIDENT`** — a missed gate, skipped required check, wrong review boundary, stale/false state, unrecorded deviation, review-timing mistake, or evidence/finding sync failure sets state status `PROTOCOL_INCIDENT`: **new build cards BLOCKED** until the current-stage checklist re-runs and the incident row records cause + repair. Allowed while open: boot check · rerun stage checks · update state/handoff · repair current card · open findings · stop-action card. Forbidden while open: start new build card · claim module done · request cross-family · final-ready. **Scope: PROTOCOL corrections only** — a CEO product/design rejection is a product finding, NOT an incident, unless it reveals a skipped gate or false state. This replaces "patch the named symptom and continue".
*✅ BUILT + biting 2026-06-12: `cx check state` validates protocol_deviations rows, review_boundary enums (block REQUIRED at session-start in build modes), ceo_deferred refs, and the incident row; `cx check card --state` blocks non-repair build cards and cross-family/final-ready requests while an incident is open; `cx check final-ready` rejects an open incident — clauses STATE-REVIEW-BOUNDARY-* · STATE-CEO-DEFERRED-NEEDS-REF · STATE-DEVIATION-TYPED · STATE-INCIDENT-CAUSE-REPAIR · CARD-INCIDENT-BLOCKS-* · FINAL-READY-BLOCKED-BY-INCIDENT.*

## The module_acceptance gate family — the per-module Andon hard wall (V1.10, folded 2026-06-13)
V1.10 re-anchors V1 to its own charter (§3 "module build → CEO approves → next module"): every module ends at a **hard-wall Andon stop** — the build **mechanically cannot start module N+1** until the CEO has *seen the running module N* and a machine receipt records `accepted`. The wall is the **checker**, never the builder's goodwill. The family rests on a **frozen module registry**, and folds as ONE cohesive set (registry → module-start → Andon receipt → conformance → regression → quality card), not loose clauses.
- **MODULE-REGISTRY (frozen at G1 compilation)** — the build's modules are a frozen, ordered `module_registry` (`MODULE-REGISTRY.template.yaml`): each `module_id` · the requirement IDs it covers · `risk_flags` (money/login/data/ui) · `dependency_modules` · the card IDs that build it · `expected_acceptance_artifacts` — all frozen INSIDE the packet hash: the registry file lives in the frozen packet dir, so its bytes are part of the packet content hash (a non-empty `frozen_packet_hash` marks it frozen; the binding to the cards' `locked_packet_hash` is content-deep — see MODULE-START). Every build card carries its `module_id`. Without the registry there is no "module N+1" to gate. [RULE:module-registry]
- **MODULE-START (the order wall, P0)** — `cx check module-start <card> --state --registry [--repo-root]`: a module-advancing card (`MODULE_BUILD`/`MODE_A_UI`) is authorized ONLY when its `module_id` is in the registry AND every **prior required** module — its `dependency_modules` PLUS every module ordered before it (the locked-plan order is itself a dependency) — is **validly accepted** in state. "Validly accepted" RE-VALIDATES each prior module's bound receipt through the shared acceptance validator (`validate_accepted_module`) — it never trusts the `accepted_modules` id set, so a hand-authored `accepted` with no bound receipt does NOT unlock the next module. An out-of-registry card, or any prior module not validly accepted, = **P0 block**. The registry must be the FROZEN one bound to the card by **CONTENT, not by a self-declared string** (V1.10 R4 — closes the R3 open P0): `module-start` takes `--packet-dir` (the frozen packet dir, which CONTAINS the registry + manifest), RE-HASHES it with the deck's packet-hash recipe, and requires the recompute equal the card's `source_map.locked_packet_hash`; the module order is read ONLY from the **canonical `<packet-dir>/MODULE-REGISTRY.yaml`** (`--registry`, if given, must resolve to exactly that path). Trimming a module from the canonical registry changes the recompute → **P0 block**; pointing `--registry` at an external registry, OR at an ALTERNATE registry-shaped file committed inside the packet (registry IDENTITY, not merely "inside the packet" — GPT R4), → **P0 block**. The frozen packet must be **self-contained — no symlinks**: a symlink anywhere under the packet, the canonical registry itself, the packet root dir passed as `--packet-dir`, OR any ANCESTOR component between `--repo-root` and the packet, is rejected (P0), because following it would point the packet at content OUTSIDE the repo (GPT R5/R6/R7). The registry order must also be **unambiguous**: a duplicate `module_id` (which would let the wall pick the first occurrence and skip a real prior), or a `dependency_modules` entry naming an unregistered module, is rejected (P0). (Requiring the recompute also equal the registry's own `frozen_packet_hash` is impossible — a registry inside the packet cannot store the hash of the packet that contains it; content-binding to the card's hash + the inside-packet rule is strictly stronger than the old string equality.) **Wired into the every-card rail:** `cx check build-turn` runs `module-start` for every `MODULE_BUILD`/`MODE_A_UI` card (packet dir via `state.packet_dir`, registry via `state.module_registry_ref` — repo-relative paths only, absolute/`..` rejected; either absent = fail-closed), so the gate family bites on the normal build path — not opt-in / out-of-band. `cx check card` rejects a build-mode card with no `module_id` (P0). Closes the omit-a-module / start-an-unknown-card / opt-in-rail holes. [RULE:module-start-order-wall]
- **ANDON-WALL (machine receipt, not goodwill — P0)** — `cx check module-acceptance --module-id --state --acceptance`: acceptance is a **`MODULE-ACCEPTANCE.yaml` receipt** (generated by `/cx-accept` once that runner exists; hand-authorable until then — see HONEST SCOPE below) — verdict `accepted`, generator-stamped (`generated_by`, enforced as non-empty today), with binding fields present (`state_sha_before` + `quality_card_hash`), and the receipt **FILE bound to state by sha12**. The check refuses to unlock module N+1 unless that bound acceptance receipt exists; a hand-edited `accepted` with no receipt, a swapped/mismatched-hash receipt, an empty `generated_by`, or a non-`accepted` verdict = **P0 block**. State stores only the compact `accepted_modules[]` ref/hash, never the blob (tiny-state rule). **The builder's word is never the wall — the bound receipt is.** The receipt must also **identify its module** (non-empty `module_id` exactly equal to the requested module — a module-less bound blob would accept ANY module), and its `acceptance_ref` must be a **repo-relative, non-symlink path inside the repo** (no absolute / `..` / symlink / outside-repo ref — else the wall would read arbitrary external bytes as the receipt) (V1.10 R4, GPT R10/R11). **HONEST SCOPE (V1.10):** the receipt-file↔state sha12 binding is enforced now (a swapped or hand-edited recorded acceptance is rejected); the receipt's *internal* `state_sha_before`/`quality_card_hash` are presence-asserted until `/cx-accept` RECOMPUTES them against the real prior state + quality-card artifact (full boot-parity recompute = acceptance-binding debt, blocked on the deferred runner per CHARTER §4). [RULE:module-acceptance-receipt]
- **DESIGN-FIDELITY-BLOCKS (P0, new builds)** — `cx check design-fidelity` is P0-blocking for new UI builds: no audited + CEO-accepted `ui_lock_manifest` (audit_status PASS + ceo_acceptance_ref) → no UI build. The legacy `ui_marker_manifest` advisory-WARN escape is removed for new builds; `--legacy-migration` keeps it ONLY for an in-flight project's documented migration debt, until its next clean module boundary.
- **CONFORMANCE-TO-LOCK (P1, the non-visual anti-drift / GAP-1)** — `cx check module-quality`: a `risk_flags` money/login/data module's acceptance must attach an **extracted-actuals** diff (built code vs the locked `IMPLEMENTATION-CONTRACT` manifest — routes · schema · API · functions · business rules · structure) in `conformance_evidence_refs`; a free-text conformance answer alone = P1. design-fidelity blocks *visual* drift; this blocks *architecture/logic/data-shape* drift (a real failure class from prior builds).
- **REGRESSION-SMOKE + QUALITY-CARD (P1)** — `cx check module-quality --acceptance --registry --module-id`: every acceptance carries a quality card answering the **core four** (security · efficient · regression · tests) + a conformance answer; a module touching the shared shell/routes must show a **PASS regression-smoke** over the already-accepted modules. Missing card, or a non-PASS shared-shell smoke, = P1. **`--registry` is REQUIRED (fail-closed):** the risk source (money/login/data + shared-shell) is the frozen registry, never the acceptance receipt's own self-declaration — omitting it would let a money module silently skip conformance/regression.
- **DEFECT LEDGER (post-ship, Q2)** — a defect the CEO finds AFTER ship is recorded as an `open_findings` row flagged `found_post_ship: true` + `found_by: ceo` (NOT a parallel ledger file — it reuses the existing `caught_by`/`should_have_been_caught_by` schema). A `found_post_ship` row must name `found_by` (P2). Each → a fix module + a candidate Kaizen lesson (which gate *should* have caught it). An OPEN post-ship P0–P2 = not shippable (final-ready's all-zero rule counts it).
- **PHANTOM-COMPLETION GUARD (PROP-028, folded v1.12 2026-06-19 — MECHANICAL)** — `cx check module-acceptance` reads the receipt's `repo_sha_before` (the repo HEAD when this module's build STARTED) and rejects an `accepted` module whose `git diff repo_sha_before..HEAD` is EMPTY — phantom completion (no real change shipped: no-op build / lost-work resume / wrong-branch checkpoint) → **P0** when `risk_flags` ∩ {money,login,auth,data,security}, else **P1**; a malformed (not hex / <12 chars) or non-ancestor baseline = P1; an absent `repo_sha_before` = P1 unless a typed `legacy_no_baseline` carve-out reason is present (then a NON-BLOCKING P2 migration-debt advisory for pre-field modules). Read-only (git name-only, never writes state or the receipt). Built at the Andon wall ONLY → final-ready is covered TRANSITIVELY (it requires every module validly accepted). **ACCEPTED LIMITATION (CEO-D-015, the LEAN version):** `repo_sha_before` is *self-declared* in the receipt, so the guard reliably catches ACCIDENTAL phantom completions but NOT a deliberately backdated baseline; fully closing it = the deferred full version (auto-record a state-bound module-start SHA). [RULE:module-acceptance-receipt]
- **LIVE SLICE DELIVERY (PROP-032, folded v1.13 2026-06-20 — the CEO drives each page live)** — re-scopes the build unit to a per-PAGE vertical slice and adds a CEO LIVE-DRIVE accept on top of this Andon wall (reuses it entirely; adds no new gate family). The frozen MODULE-REGISTRY flags a user-facing page module `live_slice: true` — the build's chunks become page-sized so each page gets the order wall + accept receipt + "no next until accepted". A `live_slice` module's MODULE-ACCEPTANCE receipt MUST carry a typed **`live_slice_accept: {live_url, ceo_drove: true, ceo_turn_ref, repo_sha}`** — proof the CEO opened + DROVE the real running build on the local machine (served locally; the local-serve pattern), not a Mode A screenshot/shell or a module-level batch. Enforcement is **P0** and reads `live_slice` ONLY from the frozen registry (the trusted source — never the receipt's self-declaration): `validate_accepted_module` (the order wall `cx check module-start` re-validates every prior module through it) rejects a live_slice prior with no valid `live_slice_accept` block → **the next slice mechanically cannot start** ("no accept, no next"); `cx check module-quality` enforces the same block so the LAST slice (no next to gate) is covered too. The "real engine, not a shell" half is the EXISTING G3 no-`DESIGN_FIXTURE`-in-a-built-module rule (past Mode A) — PROP-032 relies on it, does not re-build it. **WALKING SKELETON FIRST:** slice 0 = `walking_skeleton: true` = the shell + navigation with all pages present but empty (tappable from day one) — EXEMPT from the real-engine bar (it is meant to be empty), but it STILL needs the live-drive accept (the CEO taps through the running skeleton). **GRANULARITY:** a live_slice must stand alone (independently runnable + acceptable); a page that cannot is carried by slice 0 (the shared shell). **HONEST SCOPE:** the checker proves the running build was DECLARED (live_url) and the CEO RECORDED driving it (typed receipt); it RELIES on the builder honoring G3's no-`DESIGN_FIXTURE` rule for the "real engine" half — it does NOT independently re-scan the built module for a surviving fixture — and it never claims the experience FELT right (that stays the CEO's accept; live_url is not network-probed — over-claiming would be the cardinal sin). The order wall content-binds the registry + robustly gates every live_slice WITH a successor; the LAST slice's live-drive rides `cx check module-quality`, which trusts its `--registry`/`--acceptance` the same way it trusts them for risk/conformance (a wrong-module receipt is rejected by the receipt↔module-id match; a doctored registry is the pre-existing module-quality trust assumption; final-ready does not re-check live_slice — a documented residual). Lock = experience + contract frozen: an accepted slice is frozen by the existing accepted-module + order wall; changing a locked slice uses the EXISTING unlock protocol. [RULE:module-acceptance-receipt]
- **VERIFY-APP GATE (PROP-036, fold v1.17 2026-06-24 — runtime behavior is machine-checked before the CEO ever drives it)** — PROP-032 proves the CEO DROVE the running build, but the CEO is still the *first* thing to exercise the screen at runtime: a screen can be built, green, and live-drive-offered while its behavior was never machine-exercised (a live-production module shipped a 617px screen that passed 37/37 green gates; the analog for *behavior* is a flow that green tests never clicked through). The **verify-app gate** closes that: a `live_slice` module's MODULE-ACCEPTANCE receipt MUST carry a typed **`verify_app: {passed: true, repo_sha, generated_by, criteria_ref}`** — proof the **verify-app agent DROVE the running build and checked its acceptance criteria at runtime**, so a live_slice's behavior is **machine-verified before the CEO live-drive**. It is a **PRECONDITION** to the `live_slice_accept` above: `validate_verify_app` runs INSIDE `validate_live_slice_accept` (the single chokepoint shared by the order wall `cx check module-start` AND `cx check module-quality`), so a live_slice module cannot be validly accepted — and the next slice mechanically cannot start — unless a passing `verify_app receipt` exists. The build-turn additionally offers an **OPT-IN early-catch** (`cx check verify-app` when a card declares `verify_app_ref`; else NOT_APPLICABLE) so a malformed/forged/failing receipt is caught at slice completion, before the screen is surfaced — but the MECHANICAL "every live_slice, once" guarantee is the module-acceptance precondition above, NOT this opt-in step (a completion card that omits `verify_app_ref` is still caught by the wall). **Enforcement:** a missing / forged / unbound (non-hex-shaped `repo_sha`) `verify_app receipt` = **P0**; an honest `passed != true` (the agent drove it and the runtime check failed; `passed` must be a real boolean true, not a `"true"`/`"yes"` string) = **P1** — the slice is not accepted and the live-drive is not offered until the behavior passes. **HONEST SCOPE:** the checker proves a generator-stamped receipt carrying a hex-SHAPED `repo_sha` recorded a PASS — it does NOT verify `repo_sha` is HEAD / a real commit (shape only, mirroring `live_slice_accept.repo_sha`; a stale/forged valid-hex receipt passes shape), does NOT itself re-run the app (the agent's job), and never claims the behavior was perfect (the CEO live-drive still judges the experience; over-claiming would be the cardinal sin). The verify-app gate is **BLOCKING, never advisory**, and rides the existing live_slice + module_acceptance machinery — no new gate family, no change to the G1–G8 order. [RULE:verify-app-gate]

## Review methodology — how EVERY model review is ASKED (PROP-006, folded at V1 lock 2026-06-10; re-scoped by PROP-015 2026-06-10)
Every model review dispatch — SELF-REVIEW and cross-family alike, from the **FIRST**, not only re-reviews; reviews of docs, packets, cards, code, and protocol changes alike — asks for three things, never just "find problems" / "fix or approve": [RULE:review-three-leg-ask]
1. **CONTINUITY** (all reviews): if a prior review arc exists, carry it in full so the reviewer builds a cumulative picture, not an isolated pass; if this is the **first review**, say "first review" and check fit against the architecture + locked decisions.
2. **PROBLEMS**: new findings, by severity (P0–P3).
3. **APPROACH / fresh problem-solving**: "how would YOU do this; what would you do differently and WHY is it better; what are we not seeing; what's the right structural move?" — get the divergent perspective from the START (cheapest before rounds are sunk), not as an afterthought.

**Paste-ready three-leg dispatch stanza (PROP-015)** — copy VERBATIM into every review dispatch prompt; never re-compose from memory:

```
Return three explicit sections:
1. CONTINUITY — given the prior review arc supplied above (or "first review" if none):
```

## G8 · Final-ready gate
Nothing ships with any P0/P1/P2/P3 open or any known issue (see `SEVERITY.md`). The `FINAL-READY-CERTIFICATE` is **auto-assembled by `cx check final-ready`** from evidence already collected (capsules current · regressions pass · CEO approvals present · security closeout done · recovery proof) — never hand-written. Verdict `READY` only when every line passes. The final-ready PROCESS also re-runs `cx check deck` (packet coverage must still hold at ship); folding deck into the `final-ready` command itself = Kaizen/future — honest scope.
**Final cross-family ship-gate (V1.10, P0):** `cx check final-ready [--repo-root]` also requires a bound `final_cross_family_receipt` — the final whole-build cross-family review (opposite family of the builder). final-ready **opens the `receipt` file** (in-tree path only — absolute / `..` / symlink escape = P0), recomputes its sha12 against `receipt_hash` (fabricated/stale hash = P0; mirrors the Andon binding), and requires it to be a **TYPED review artifact** (`final_cross_family_review` with `reviewer_family`/`builder_family`/`verdict`) whose fields **MATCH the state claim** — state asserting `PASS` over a receipt that says `REJECTED` = P0 (a bound blob of arbitrary bytes is not a review). It also requires `reviewer_family ≠ builder_family` and the `verdict` **present** and `PASS`/`FIX_FIRST_RESOLVED` (a *missing* verdict is not a silent pass = P0). Cross-family is the LAST review = the ship gate (the CEO's Option-B design); shipping without its bound receipt = **P0 block** ("last" can never become "never"). A budget block must be a typed `protocol_deviation`/STOP that saves state (PROP-020), never a silent skip.

**Built-App Audit — pre-G8 precondition (v1.12, P0):** final-ready requires a completed Built-App Audit (see `BUILT-APP-AUDIT.md`). The audit is a final, read-only, whole-app reconciliation across 3 angles — requirements coverage · original CEO asks · shipped reality — including the **killer check**: for every shippable feature, a real production caller/trigger (route/schedule/cron/entrypoint) must exist; zero non-test callers = flagged (built + tested + green ≠ wired and running). An app **cannot** be certified final-ready until the audit has run **and** every finding is either **fixed** (Code-X card + opposite-family review — never a hot patch) or **explicitly CEO-deferred** with a recorded decision in `CODE-X-STATE.yaml`. `cx check final-ready` enforces this mechanically: state must carry a typed `built_app_audit` block with `status: run`, `findings_dispositioned: true`, and a repo-relative non-symlink `report_ref` — any of these missing or false = P0 block. This complements (does not replace) the existing G8 / `FINAL-READY-CERTIFICATE`.

**Always-on production-readiness for client-facing ships (PROP-014 Piece 4 — `PACKET-CONTENTS.md`):** R1 resilience/failure-mode (timeouts · clear error states · no duplicate-on-retry) · R2 real-environment proof (fresh install · empty states · env-completeness) · R3 observability wired BEFORE ship (crash + uptime + usage; first-hour watch) · R4 recovery runbook (plain "if X breaks do Y" + 1-command rollback a non-coder can run). Risk-gated R5 load / R6 cost caps fire from the trigger checklist answered at planning (coverage-map row 19); S1 dependency-scan is a HARD G7 gate (PROP-027, `cx check dep-scan`); S2–S4 ride the existing security reviews. Skips are written, never silent.
