## When to run
On a protocol change · on V1 first creation · before a project's first build under V1 · before declaring V1 stable. **Not** on every module build.

## Growth rule (Kaizen / anti-bloat) [RULE:eval-count]
The set grows **slowly** — one new eval per **new failure-class** learned (NOT one per lesson), each a tiny protocol self-test. If evals grow heavy, **consolidate** — this 022-E slim IS that consolidation. A new failure that slips through becomes: postmortem → lesson → (if it's a new class) a new eval row below.
