# Code-X V1 — checkers/ : CX-CHECK SPEC

> Level-B mechanical enforcement: ONE command surface `cx check` with subcommands. Tiny, single-purpose, fast (< seconds), each independently deletable. NOT an orchestrator (no `cx run` / `orchestrate` / `auto-route` / `self-mutate`).
>
> **Contract rule:** a check must ENFORCE exactly what it claims — never green while not biting. A non-coder CEO reads a green check as "the gate fired." Over-claiming is the cardinal sin (GPT 5.5 Pro cross-family review, 2026-06-09, found 9 cases of it; this spec is the corrected contract).

## Subcommands (each returns PASS / FIX-FIRST + findings)

- **`cx check card <card> [--state CODE-X-STATE.yaml]`** — validates a work-order's SHAPE and that its safety fields are REAL, not paperwork:
  - every field present; allowed/forbidden files + ops declared; `model_tier` named; objective one sentence;
  - **read-budget** under the kernel ceiling — use `read.estimate_tokens` when set, else the file-count cap;
  - **traceability (G1):** `source_map` present AND complete — `locked_packet_id`, `locked_packet_hash` (frozen packet hash), and each `source_sections[].{file,section,requirement_ids}`; `card_compilation.compiled_by` + `audited_by` set and **`audit_status == PASS`** (a card whose audit is `PENDING`/`FIX_FIRST` is REJECTED); [RULE:source-map-required]
  - **cross-family:** `actor_record.cross_review.family != executor.family` — same-family REJECTED unless `family_substituted: yes` + `ceo_authorization_ref`, which opens a final-ready-blocking `CROSS_FAMILY_RECHECK_PENDING`. With `--state`, the checker VERIFIES that pending item actually exists in `open_findings.items` (owner_card set); without `--state` it emits the requirement as a finding;
  - **foundation checkpoint (G5):** if `foundation_checkpoint_required: yes`, the card must carry `foundation_checkpoint_reason`, and dependent cards are blocked until its checkpoint passes;
  - `family_note` injected. *(Backs GATES G1. These fields FAIL the card if missing — not optional.)*
  - **engine routing (PROP-013, `--profiles <path>` default = canonical `BUILD-ENGINE-PROFILES.yaml`; `CX_PROFILES` env is TEST-ONLY — honored solely with `CODE_X_TEST_MODE=1`, else REFUSED P1 fail-loud so a stale env var can never silently redirect production validation — PROP-014 fold/GPT P1-02):** `execution` block present (P0 — compiler injection, never hand-written) · `execution.profiles_hash` matches the current profiles sha256-12 (P1 — stale = re-inject) · each engine branch's model/effort exists in the profiles for `execution.role` (P1 — no actor invents its own seat) · **mini/spark/haiku-class model in an implementer seat = P0** (code_writing_floor, family-neutral) · `execution.limits` ≤ profiles caps (P1) · `model_tier: top` or execution effort high/xhigh/max requires `top_allowed_reason` (P1 — ROUTING rule, now mechanical) · **CodeRabbit can never be recorded as a SATISFIED cross-family checkpoint** (P0; allowed only as a `family_substituted` stand-in with `ceo_authorization_ref` = an IOU opening `CROSS_FAMILY_RECHECK_PENDING`).

  - **review-dispatch three-leg ask (PROP-015):** when `review_dispatch.required: yes`, ALL three `three_leg_ask` fields — `continuity`, `problems`, `approach_improvement` — must be non-placeholder (rejected values: `asked`, `yes`, `included`, `N/A`, blank) AND each must contain its canonical label (`CONTINUITY`, `PROBLEMS`, `APPROACH/IMPROVEMENT`) or the card must carry a non-empty `prompt_ref` to the dispatch-prompt artifact (`prompt_ref` recommended always; relaxed per CEO 2026-06-10 — no prompt hash required); `review_kind` must be one of `SELF_REVIEW | CROSS_FAMILY | CARD_AUDIT | FINAL_READY | PROTOCOL_CHANGE`; a card in mode `REVIEW`/`FINAL_READY` MUST carry `review_dispatch.required: yes` (omitting the block is not an exemption); ordinary build cards with `review_dispatch.required: no` (or no block) are exempt → violations P1 FIX-FIRST. HONEST SCOPE: covers review-dispatch work-orders only — planning-stage free-text review prompts stay process-enforced via the GATES paste-ready stanza. [RULE:review-three-leg-ask]

- **`cx check state <state>`** — `CODE-X-STATE.yaml` well-formed; `protocol_stamp == Code-X V1`; exactly one `current_card`; `last_commit` present & non-empty (the anti-chat-only-drift anchor); `open_findings.counts` match `open_findings.items[]`; an open `CROSS_FAMILY_RECHECK_PENDING` blocks; **engine routing (PROP-013):** `active_build_engine` present + in `CLAUDE_CODE|CODEX_APP` (P1) · `orchestrator_model` must not exceed the `BUILD-ENGINE-PROFILES` orchestrator seat for the active engine + stage without a `top_allowed_reason` (P1; unparseable launch string = fail closed). *(HONEST SCOPE — validates the state file IN ISOLATION: it does NOT read referenced card files. Verifying each referenced card's `actor_record` and the full derived who-did-what ledger are Kaizen/future items — NOT enforced today. The git continuity check DOES exist, as the opt-in `--session-start` mode below. State the limit, don't fake the feature.)*
  - **`--session-start --repo-root <dir>`** — the git/worktree continuity guard (run at session/card START, not as a mid-card loop blocker): `state.last_commit` must be an ANCESTOR of HEAD — equality counts; a foreign-lineage or unknown sha = P1 ("state points outside this branch's history — wrong worktree/branch or stale state") · a dirty tree requires `wip_continuation.marked: yes` in the state (P1), and a marked WIP missing `owner_card`/`handoff_ref` = P2 ("WIP marked but unowned") · ancestor but >3 commits behind HEAD = an advisory `WARN:` line ONLY (no finding, exit unchanged — hard gates stay hard, soft signals stay soft). One active project per working tree — a parallel project gets its OWN git worktree (uncommitted work on a shared tree is one branch-switch from gone).
  - **builder-standard acknowledgment (PROP-014 Piece 5):** in a build mode (`MODE_A_UI | MODULE_BUILD | FIX`), `session_start.builder_standard_read` must exist with `status: PASS` + `file` + `hash` (P2 when missing — a build session acknowledges `BUILDER-STANDARD.md` at session start); when the named file resolves under `--repo-root`, a recorded hash that no longer matches the live file = P2 ("standard drifted since acknowledgment"). HONEST SCOPE: proves WHICH version was acknowledged — not that it was internalized. [RULE:builder-standard-session-read]
  - **lessons-preload acknowledgment (PROP-017):** in stage `PLANNING_STUDIO`, `session_start.lessons_preload` must exist with `status: PASS` + `file` + `hash` (P2 when missing — a planning session reads the ACTIVE stage-planning lessons from `MEMORY/LESSONS.yaml` once at session start); same drift rule as the builder-standard ack when the named file resolves. Planning-sessions-only — the build read path is untouched. HONEST SCOPE: proves WHICH lessons file was preloaded — never that lessons were applied; the two-pass anti-anchoring rule stays a model/process discipline. [RULE:lessons-preload-ack]

- **`cx check deck <cards-dir> <packet-dir> [--manifest <path>]`** — REVERSE coverage at the G1 compile boundary. Forward traceability (`cx check card`'s `source_map`) proves each card came from the packet; this proves the DECK covers the PACKET — a requirement cannot silently vanish at compile: [RULE:deck-reverse-coverage]
  - the `requirements-manifest.yaml` (default `<packet-dir>/requirements-manifest.yaml`; `--manifest` must still resolve INSIDE the packet dir — absolute/`..` escapes rejected) declares every requirement id with a disposition `BUILDING | NOT_BUILDING | NOT_APPLICABLE | CEO_DEFERRED`;
  - every BUILDING id must appear in ≥1 card's `source_map.requirement_ids` (P0 — "dropped at compile") · NOT_BUILDING / CEO_DEFERRED require `ceo_decision_ref` (P1 — dropping or deferring an approved requirement is ALWAYS a CEO-visible decision) · **every `ceo_decision_ref` must RESOLVE to a `CEO-DECISION-LEDGER.md` row id `CEO-D-NNN` inside the packet (P1, PROP-014 — "decision lives in chat" is not a decision; legacy decisions migrate as `CEO-D-LEGACY` rows)** · NOT_APPLICABLE requires `reason` (P1) · a card claiming an id the manifest doesn't know = ghost requirement (P1) · missing/malformed/empty manifest = P1;
  - **frozen-hash binding:** the packet hash is RECOMPUTED — pinned recipe: sha256 over every regular file under the packet dir, sorted by POSIX relpath, each contributing `relpath + NUL + bytes` — and must equal EVERY card's `source_map.locked_packet_hash` (P0). The manifest lives inside the packet dir, so it is inside the hash: editing packet OR manifest after the deck is cut breaks the deck. A frozen packet must be **self-contained — no symlinks**: any symlink under the packet dir, OR the packet root dir itself, raises (V1.10 R4, GPT R5/R6), because following it would hash the symlink target's bytes (content from OUTSIDE the packet); module-start surfaces this as P0, the deck as P1;
  - PASS output lists every dispositioned-out requirement with its disposition (CEO-visible — the manifest can never become a silent trash bin). Runs at G1 (deck compiled) and re-runs as a final-ready process step (G8).

- **`cx check packet <packet-dir>`** — the packet-contents floor (PROP-014, mechanical half of the completeness-audit gate; canon `PACKET-CONTENTS.md`): [RULE:packet-contents-floor]
  - `coverage-map.yaml` inside the packet dir; all 20 category rows present, no duplicates; each row `DONE` + named file existing inside the packet (path-escape rejected) and non-trivial (≥200 bytes — a stub is not packet content), or `N/A` + written `na_because`; **always-on categories (1–4, 10–13, 15, 16, 19, 20) may never be N/A** (P1 — a thin packet must not pass green);
  - `requirements-manifest.yaml` present (content is `cx check deck`'s contract) · `CEO-DECISION-LEDGER.md` present with BOTH required sections (`## CEO Asks Register` + `## Decisions`) and no ask left `MISSING`/`PARTIAL` (P1 — open asks cannot be frozen over) · `completeness-audit.md` present, non-trivial, no open verdicts (P1);
  - **locked style direction (PROP-016):** when coverage category 14 (UI/workflow locked designs) is `DONE`, the packet's PRODUCT-TASTE-LOCK must carry a COMPLETE `locked_style_direction` block — `applicable: yes` + `chosen_variant_id` + `chosen_variant_path` + `accepted_by` + `accepted_at`, or `NOT_APPLICABLE` with a written `why:` (P1 — without the Style-Select outcome the taste lock is incomplete and G7 blocks; category 14 `N/A` = not user-facing = clause silent); [RULE:locked-style-direction]
  - HONEST SCOPE: this is the MECHANICAL half only — the semantic completeness audit (fresh cold reader tracing every CEO ask to its satisfying doc location) stays a model step; this check proves the artifacts exist and nothing open was frozen over, never that the content is COMPLETE. Runs before CEO validation + packet freeze (G7 line).

- **`cx check module-start` / `cx check module-acceptance` (V1.10 — the module_acceptance gate family)** — the per-module Andon hard wall. A frozen `module_registry` (`MODULE-REGISTRY.template.yaml`, compiled at G1, committed INSIDE the frozen packet dir so its bytes are part of the packet content hash) lists the ordered modules; every build card carries its `module_id`. `cx check module-start <card> --packet-dir --registry --state [--repo-root]` blocks a module-advancing card unless every prior required (`dependency_modules` ∪ earlier-ordered) module is **validly accepted** — it RE-VALIDATES each prior module's bound receipt through the shared `validate_accepted_module` (never trusts the `accepted_modules` id set, so a hand-authored `accepted` with no receipt does not unlock the next module), and rejects a card whose `module_id` is not in the registry (P0). **CONTENT-deep packet binding (V1.10 R4):** `--packet-dir` is required (fail-closed); the order wall re-hashes that packet dir (the deck's packet-hash recipe) and requires the recompute equal the card's `source_map.locked_packet_hash`, and the order is read ONLY from the **canonical `<packet-dir>/MODULE-REGISTRY.yaml`** (`--registry`, if given, must resolve to exactly that path) — so a trimmed registry that keeps a stale `frozen_packet_hash` string, an external registry, OR an alternate registry-shaped file committed inside the packet (registry IDENTITY — GPT R4), is rejected (P0; closes the R3 string-deep bypass + the R4 alternate-in-packet bypass). (Equality to the registry's own `frozen_packet_hash` is impossible — a registry inside the packet can't store its containing packet's hash; content-binding to the card hash + inside-packet is strictly stronger.) It is **wired into `cx check build-turn`** (packet dir via `state.packet_dir`, registry via `state.module_registry_ref`; either absent on a module-advancing card = fail-closed) so the order wall bites on the normal rail; `cx check card` rejects a `MODULE_BUILD`/`MODE_A_UI` card with no `module_id` (P0). `cx check module-acceptance --module-id --state --acceptance --repo-root` requires a `MODULE-ACCEPTANCE.yaml` receipt — verdict `accepted`, `generated_by` stamped, binding fields present (`state_sha_before` + `quality_card_hash`), and the receipt FILE bound to state by sha12 — before module N+1 unlocks (P0); state stores only the compact `accepted_modules[]` ref/hash, never the blob. A hand-edited `accepted` with no bound receipt, an empty `generated_by`, or a sha-mismatched/swapped receipt is rejected (P0). **HONEST SCOPE (V1.10):** the file↔state sha12 binding is enforced now; the receipt's internal `state_sha_before`/`quality_card_hash` are presence-asserted until `/cx-accept` recomputes them (acceptance-binding debt, CHARTER §4). `cx check module-quality --acceptance --registry --module-id` REQUIRES `--registry` (fail-closed — risk comes from the frozen registry, never the receipt's self-declaration). **PROP-028 (phantom-completion guard):** the receipt also carries `repo_sha_before` (the repo HEAD when the module's build STARTED); the Andon wall reads it and resolves `--repo-root` (else the state file's dir) — a malformed value (not hex / <12 chars) or a baseline that is not an ancestor of HEAD is P1, and an EMPTY `git diff --name-only repo_sha_before HEAD` is phantom completion (no real change shipped) → P0 when the module's `risk_flags` intersect {money,login,auth,data,security}, else P1; an absent `repo_sha_before` is P1 unless a typed `legacy_no_baseline` carve-out reason is present (then a non-blocking P2 migration-debt advisory). The checker only VALIDATES this baseline (read-only); it never writes state or the receipt. [RULE:module-registry] [RULE:module-acceptance-receipt]

- **`cx check scope <card> <diff>`** — the real diff touched ONLY `allowed_files` and NO `forbidden_files`:
  - **empty `allowed_files` with a non-empty diff FAILS** (closes the "edit anything not forbidden" hole) unless `mode: REVIEW` with no edits expected;
  - path matching uses real glob (`fnmatch`; treat `/**` as directory-prefix recursion) — NOT substring;
  - **all SEVEN `security_tripwire` fields are verified against the real diff** (auth · secrets · money/balances · bank/PII · upload/restore/import · network/public-surface · logs/error-output): a field claimed `no` while the diff touches its pattern set = FAIL. No unused pattern lists. *(Closes the self-attestation hole — backs G2 tripwire.)*

- **`cx check evidence <card>`** — every `evidence_required` path exists, is non-empty, no faked-pass patterns; **paths resolve relative to the card/project root, not the CWD** (reproducible regardless of where `cx` runs). **(FIX cards) diff-aware test-edit guard:** a test file edited in the diff is allowed ONLY when `fix_test_edits.allowed: yes` + `reason` + the file is listed in `touched_test_files` with `regression_test_evidence`; otherwise FAIL. **a NEW regression test is allowed only** with that explicit authorisation — fix cards may never **weaken**/delete/score-game tests. [RULE:fix-card-test-edit]
  - **Proof-card honesty (PROP-022, built 2026-06-12):** when the card carries `evidence_claims`, each row must be fully typed (`claim_id, claimed_verdict, command, exit_code, log_path, produced_at, produced_by` — missing field P1) and the checker **RE-READS the log** (resolved relative to the card dir): missing log = **P0** · claimed PASS with nonzero exit and no declared `nonzero_pass_semantics` = **P0** · an `expect_contains` marker absent from the log or a `fail_markers` entry present in it = **P0** claim/log mismatch. **Scoped positive control:** `claim_type ∈ design_fidelity · interaction/click/swipe · source_truth · money_invariant · security_boundary · import/dedupe/recovery/destructive` (or `sole_proof_of_high_risk_claim: yes`) requires a `positive_control` red run — missing block = P1; its log missing = P0; a "red" run that does not demonstrate the EXPECTED failure (`expected_failure_marker` absent / exit 0) = P1. **Event modality:** interaction-class claims need the typed `modality` block (`handler_event_family, dispatched_event_family, modality_source, event_reaches_handler_evidence`, families ∈ touch|pointer|mouse|keyboard); dispatched ≠ handler = **hard P1**. **Baseline pinning:** a claim comparing output to a reference (`claim_type: design_fidelity` or `compares_to_reference: yes`) must pin `baseline {baseline_ref, baseline_hash, baseline_source}` = P0 when missing; `baseline_source` naming builder same-wave output = **P0** (invalid as "expected").

- **`cx check cost`** — append/validate the `WORK-ORDER-COST-LOG`; roll up the waste alarm. **`review_fix_cycles > 1` = waste/FAIL** (one-and-done discipline), tracked SEPARATELY from bounded `self_heal_attempts` (technical auto-retry, allowed within its cap — a different loop). Flags `over_read / wrong_model_tier / repeated_review / loop`.
  - **SELF_REVIEW cap (V1.10, A3 exception):** `cx check card` permits `loop_budget.review_fix_cycles ≤ 3` ONLY for a `review_dispatch.review_kind: SELF_REVIEW` card (the bounded model-escalation loop — builder → stronger → strongest); every other card (MODULE_BUILD, CROSS_FAMILY, CARD_AUDIT, FINAL_READY) stays one-and-done (`≤ 1`). The cost-log one-and-done waste flag above is unchanged.

- **`cx check final-ready <state>`** — AUTO-ASSEMBLE the `FINAL-READY-CERTIFICATE`; verdict `READY` only if EVERY line below is an explicit `PASS` read from existing evidence/state — a MISSING or non-`PASS` field = **NOT READY** (never default-pass):
  - `open_findings` count zero AND `items` empty (incl. no `CROSS_FAMILY_RECHECK_PENDING`);
  - `module_capsules_current: PASS` · `module_regressions_pass: PASS` · `ceo_module_approvals_complete: PASS` · `security_closeout: PASS` · `recovery_proof: PASS`.
  - **final cross-family ship-gate (V1.10):** a bound `final_cross_family_receipt` — final-ready OPENS the `receipt` file and recomputes its sha12 vs `receipt_hash` (fake path / fabricated-or-stale hash = P0), requires `reviewer_family ≠ builder_family`, and requires `verdict` PRESENT and `PASS`/`FIX_FIRST_RESOLVED` (a MISSING verdict is not a silent pass = P0). `[--repo-root]` resolves the receipt path (default: the state file's dir).
  - **Built-App Audit precondition (v1.12):** a `built_app_audit` block with `status: run` + `findings_dispositioned: true` + `report_ref` that is repo-relative, non-symlink, **exists**, is a **directory**, and contains **`AUDIT-SUMMARY.md`** — a missing block, a non-`run` status, undispositioned findings, or a `report_ref` that is absent / not a dir / has no summary file = P0 (LIGHT only: existence + type + summary-file + path-safety, never report parsing). See `BUILT-APP-AUDIT.md`.
  - It may only READ / assemble / check evidence that already exists — it must NOT run builds, route actors, edit source, or create missing evidence (that would make it a runner). *(Backs GATES G8.)*

- **`cx check consistency [--strict] [--registry <path>]`** — every registry rule's canonical phrases appear (verbatim, or a `[RULE:id]` pointer) in each `appears_in` file; scans BOTH `.md` and `.yaml`.
  - **HONEST LIMIT (do not over-claim):** this catches **missing / deleted / stale** rule phrases. It does **NOT** catch a meaning-FLIP that keeps the phrases but negates them. A per-rule `banned_negations` list is a CHEAP tripwire for obvious reversals that assert a gate is no longer enforced or a requirement has been dropped — useful but **not complete**. **Any meaning-flip-sensitive rule still requires a model review when changed.**
  - **Sweep scope (PROP-005/012, folded 2026-06-11):** the key-phrase sweep covers **rule-bearing canon only** — `scan_scope.rule_bearing` in the registry ∪ every `appears_in` path (registering a rule auto-pulls its files in; a registry without `scan_scope` keeps the legacy full-tree sweep). Structural exemptions, never swept: the registry itself + `checkers/tests/`. Out-of-scope narrative/history is **silent in both modes** — narrative cites rules, canon defines them; a stale copy there goes undetected BY DESIGN. **Sweep silence never exempts an ACTIVE dispatch artifact** — the current card/deck/state are checked by their own `cx` commands (G1/G7 path), which is where operational drift bites. In-scope message split: full canonical present → "register it in appears_in" (duplication); partial → "possible unregistered/paraphrased copy".
  - **`--strict`** (the protocol-change consistency gate): an in-scope file NOT in `appears_in` that carries a rule's key-phrases FAILS (not just WARN). With scope declared the canonical tree runs **0 WARNs / PASS** — every protocol-change session must end with `cx check consistency --strict` PASS.
  - **Path safety:** all `appears_in` / `scan_scope` / `--registry` paths are bounded to the Code-X root — reject absolute paths and `..` escape (`resolved.is_relative_to(root)`).

- **`cx check boot --state <state> --repo-root <dir> [--out <receipt>] [--handoffs-dir <dir>]`** (PROP-018, built 2026-06-12) — GENERATES the session boot receipt; the model never authors hashes (anti ack-theater). Hashes the four canon files (START-HERE/KERNEL/GATES/BUILDER-STANDARD, sha256-12, resolved relative to the cx binary's Code-X-V1 root), runs the session-start state check (minus the boot-ack clause it is about to make satisfiable), records the latest handoff path (lexically last `handoffs/*.md`) + its sha12, and BINDS the receipt to this state at this point in history: `state_sha12` (canonical sha of the state file MINUS the ack block) + `repo_head` (GPT cross-review 2026-06-12 — forging a receipt is equivalent to running the command). Receipt default: `protocol-boot-receipt.yaml` next to the state file; output prints the exact `session_start.protocol_boot_ack` block to copy (receipt path + receipt sha12). Exit 1 when the state check fails — the receipt is still written, recording the failure; `cx check state` rejects an ack of a failed receipt. State-side enforcement (build modes, `--session-start`): missing/stale/tampered ack, canon drift since boot, a FAILED receipt, `generated_by` not 'cx check boot', a receipt outside the state dir / repo root, `state_sha12` not matching the live state-minus-ack, or `repo_head` not matching live HEAD = **P1**.
- **`cx check build-turn <card> --state <state> --repo-root <dir> [--diff <path>]`** (PROP-018) — the every-card aggregate gate, one PASS line per sub-check, any FAIL blocks the turn: card (with `--state`, so the incident gate rides along) · **module-start** (V1.10 order wall — for `MODULE_BUILD`/`MODE_A_UI` cards; content-deep — packet dir via `state.packet_dir` + registry via `state.module_registry_ref`, either absent = fail-closed) · scope (explicit `--diff`, else a deterministic derived list: `git diff --name-only HEAD` + untracked) · stale-allowed-files (every `allowed_files` entry exists under the repo root, matches as a glob, or is declared in the card's `new_outputs` list) · evidence · `consistency --strict` · project tests · CodeRabbit receipt where the card requires it (`coderabbit.required: yes` → `coderabbit.receipt` must exist). **CHECK/RECEIPT-ONLY, never a runner:** it runs ONLY the deterministic command explicitly named by `card.test_command` / `state.project_test_command` (one argv command, parsed without a shell — no pipes/`&&`); a missing test command = **P1 "card incomplete"**, never a guess; it never decides tests, routes actors, calls model review, creates evidence, or edits source.
- **`cx check close-turn --state <state> --handoff <path> --repo-root <dir>`** (PROP-018) — the turn-end gate. The handoff must carry ONE fenced yaml block with a typed `close_turn` mapping: `findings_delta` (rows `{id, severity P0–P3, status OPEN|CLOSED|DISCONFIRMED, finding, state_item_ref}` — ALL FIVE row fields enforced, an explicit `state_item_ref` is the reconcile key (defaults are not reconciliation); `[]` allowed, absence is not) · `evidence_paths` (each must exist under the repo root) · `next_prompt` (inline text, or a `.md` path that must exist) · `vault_sync` (`status: PASS | SKIPPED_WITH_REASON | NOT_APPLICABLE`; not-PASS requires `reason` + `where_saved`). Issue-sync is TYPED, never free-text grep: every OPEN delta row must match an OPEN `open_findings` item in state; every CLOSED/DISCONFIRMED row must NOT be open in state. HEAD's commit message must carry `Code-X-Provenance:`. HONEST SCOPE: reconciliation is delta→state (kills the state-zero/handoff-nonzero scar); it does not prove every standing state item was re-listed in the handoff.
- **`cx check evals [--checkers-dir <dir>]`** (PROP-018) — env-wrapped protocol suites: `tests/run.py` + `tests/run_contracts.py` + `cx check consistency --strict`, run with the same interpreter + environment already running cx (no PYTHONPATH/interpreter re-discovery). `--checkers-dir` redirects the two suites (TEST-ONLY — how the contract harness proves this command bites without recursing into itself); consistency always runs against the REAL canon.
- **`cx check design-fidelity --manifest <yaml> --dom <html> --screenshot <png>`** (PROP-019) — deterministic UI-fidelity gate before any model review of a UI module: the locked reference's `ui_marker_manifest` (`required_regions` by id via `id=`/`data-cx=`/`data-region=` · `required_controls` via `data-fn=` · `required_shell` selectors (`#id` / `.class` / `[attr="v"]` / literal) · `forbidden_missing`) is compared against the saved live DOM snapshot. A missing marker listed in `forbidden_missing` = **P1**, otherwise P2; a manifest with nothing deterministic to compare = P1. Marker-stuffing guard: each required control's element must be visible (no `hidden`/`display:none` stub) and carry the locked `role` + `label` (element text, aria-label, or value) — a generic template stuffed with bare `data-fn` markers cannot read green (GPT cross-review 2026-06-12). The screenshot is SUPPORTING evidence only — existence + non-empty checked (pixel-AI is never the primary gate); missing/empty = P1.
  - **`ui_lock_manifest` + lock-vocabulary coverage (PROP-022, built 2026-06-12):** the manifest root may be the extended `ui_lock_manifest` (section names `dom_markers · controls · shell_regions`; legacy `ui_marker_manifest` names still accepted) — when it is, `audit_status: PASS` + `ceo_acceptance_ref` are REQUIRED (**P0** — planning-authored, opposite-family audited, CEO-accepted, hash-frozen; the builder only CONSUMES it; no PASS = no build). Optional **`--build-vocab <yaml>`** compares the build's actual vocabulary (`translation_keys / ui_states / alert_variants`) against the lock: a build entry absent from the manifest = **P1 invented vocabulary**; a locked entry absent from the build = P2 drift. ONE owner — vocabulary coverage lives in this check, never a parallel gate.
- **Card-side PROP-019/020 clauses (inside `cx check card`):** a card whose `allowed_files` touch `templates/`, `static/`, or `*.html`, or that declares `design_fixture_purge: yes`, must carry a complete `ui_contract` (taste_lock · design_golden_master · mode_a_ceo_accept · screenshots · route_map · click_path_contract · `preservation_rule` containing "preserve"); taste lock + golden master must be IN `read.required` (P1 — named is not read). With `--state`: while a PROTOCOL_INCIDENT is open, build-mode cards other than the incident's `repair_card` = P1 and CROSS_FAMILY/FINAL_READY dispatches = P1; `cx check final-ready` rejects an open incident (P0).

- **State-side PROP-021 fix-escalation clauses (inside `cx check state`, built 2026-06-12):** `fix_cycles` rows must be fully typed (`finding_id, source_review_ref, attempt_n, seat_ref, result, review_ref` — missing field P1; attempt history is STATE-owned, never card-owned). Per `finding_id`, the ladder is strictly sequential and strictly upward: attempt ≥2 with a seat NOT above the prior attempt's (model rank, then effort rank, within the same family) = **P1 protocol drift** · an unparseable `seat_ref` on an escalated attempt = P1 fail-closed (attempt 1 may be the symbolic `original_builder_seat`) · a skipped prior attempt or duplicate `attempt_n` = P1 · a mid-ladder family switch = P1 (family alternation belongs to the 4-failure cross-family rule) · `attempt_n > 3` = P1 (fix-3 failure feeds the 4-failure rule, never a fix-4 row) · any row recording a `loop_budget_reset` = P1 (escalation changes the seat, never grants a new review cycle).

## Anti-bloat ceiling (applies to the checkers too)
Prefer adding a sub-check to an existing subcommand over adding a new script. If `cx check` grows heavy, **consolidate** (internal modules `cx_*.py` behind the one `cx` surface — no new user-facing scripts). The count is capped — a checker must earn its place.
